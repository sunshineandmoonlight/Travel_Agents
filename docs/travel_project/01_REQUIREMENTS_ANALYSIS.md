## 1. 项目概述

### 1.1 项目定位

**TravelAgents-CN** 是一个基于多Agent架构的智能旅行规划助手，帮助用户规划国际旅行。

**核心价值**：
- 像专业旅行顾问一样理解用户需求
- 自动生成详细的旅行方案
- 提供多维度评估和对比
- 支持全球任意目的地

### 1.2 项目目标

**主要目标**：
1. 为求职创建一个完整的、可展示的多Agent系统项目
2. 展示对LangGraph、多Agent协作、State管理的理解
3. 实现一个真正实用的旅行规划功能

**次要目标**：
- 学习和掌握Agent开发技能
- 积累项目经验用于面试

### 1.3 目标用户

| 用户类型 | 描述 | 核心需求 |
|---------|------|---------|
| **个人旅行者** | 计划出国旅行的个人 | 获取旅行灵感和详细规划 |
| **求职展示** | 技术面试官 | 看到复杂的多Agent系统架构 |
| **学习者** | 想学习Agent开发的人 | 参考完整的实现案例 |

## 2. 功能需求

### 2.1 核心功能流程

```
用户输入需求
    ↓
系统分析用户偏好
    ↓
推荐适合的目的地地区（多选）
    ↓
用户选择感兴趣的地区
    ↓
生成2-3个不同的旅行方案
    ↓
用户选择方案或组合
    ↓
生成详细行程和实用信息
```

### 2.2 功能模块详细说明

#### 模块1：用户需求收集

**输入项**：
| 字段 | 说明 | 必填 |
|------|------|------|
| 目的地国家 | 如"日本"、"泰国" | 是 |
| 出发日期 | YYYY-MM-DD | 是 |
| 返回日期 | YYYY-MM-DD | 是 |
| 预算范围 | 人民币金额 | 是 |
| 旅行人数 | 成人/儿童数量 | 是 |
| 兴趣标签 | 多选（景点/美食/购物/自然...） | 否 |
| 特殊需求 | 文本描述 | 否 |

**输出**：用户需求结构化数据

#### 模块2：目的地地区推荐

**功能**：根据用户需求，推荐目的地国家的适合地区

**输入**：用户需求 + 目的地国家

**处理**：
- 查询地区数据库
- 匹配用户偏好
- 计算推荐分数

**输出**：推荐地区列表（3-5个）

**示例**：
```
输入：目的地=日本，兴趣=寺庙、美食
输出：
1. 关西地区（京都、大阪） - 匹配度95%
   - 适合寺庙文化、美食丰富
2. 东京地区 - 匹配度80%
   - 现代与传统文化结合
3. 北海道地区 - 匹配度70%
   - 自然风光、海鲜美食
```

#### 模块3：多方案生成

**功能**：基于用户选择的地区，生成2-3个不同风格的旅行方案

**输入**：用户选择的地区 + 用户需求

**方案类型**：
1. **深度主题型**：针对主要兴趣深度体验
2. **均衡经典型**：经典景点 + 多样体验
3. **小众探索型**：避开人流的独特体验

**输出**：2-3个方案概要

#### 模块4：详细行程规划

**功能**：为选定的方案生成详细行程

**输入**：选定的方案 + 用户需求

**输出内容**：
- 每日详细安排（上午/下午/晚上）
- 景点介绍和游玩建议
- 交通安排（城市间、市内）
- 住宿推荐
- 餐饮推荐

#### 模块5：多维度评估

**功能**：从多个角度评估每个方案

**评估维度**：
- 体验丰富度
- 费用合理性
- 时间舒适度
- 体力消耗
- 独特性
- 可行性

#### 模块6：实用信息整合

**功能**：提供旅行所需的实用信息

**信息类型**：
- 天气预报
- 签证要求
- 汇率信息
- 安全提示
- 紧急联系方式

### 2.3 非功能需求

| 需求类型 | 要求 | 说明 |
|---------|------|------|
| 性能 | 单次规划<30秒 | 包括API调用和LLM生成 |
| 可用性 | 80%API调用成功 | 允许部分降级到演示数据 |
| 可扩展性 | 支持添加新国家 | 通过数据库扩展 |
| 可维护性 | 清晰的代码结构 | 遵循原项目规范 |

## 3. 用户交互设计

### 3.1 交互流程

```
┌─────────────────────────────────────────────────────────────┐

                            ↓

└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  系统生成方案                                                │
```

### 3.2 输出格式

**方案概要格式**：
```markdown

- 5座不同风格的古寺深度游览

⚡ 适合人群：文化爱好者、喜欢慢旅行的人
```

```markdown
- 清水寺（世界遗产）
- 祗园古街漫步

🍽️ 午餐：顺正汤豆腐
🍽️ 晚餐：先斗町小料理

🏨 住宿：京都四条河原町酒店

📝 今日提示：购买巴士一日券（500日元）
```

## 4. 系统边界

### 4.1 包含的功能

✅ 目的地国家的基础信息
✅ 主要旅游城市的景点推荐
✅ 基于用户偏好的地区推荐
✅ 多方案生成和对比
✅ 详细行程规划
✅ 预算估算
✅ 天气、签证等实用信息

### 4.2 不包含的功能

❌ 机票预订（仅提供信息）
❌ 酒店预订（仅推荐）
❌ 餐厅预订（仅推荐）
❌ 实时价格查询（使用估算）
❌ 行程中的动态调整（v1.0不支持）
❌ 社交分享功能

### 4.3 数据来源

| 数据类型 | 来源 | 更新频率 |
|---------|------|---------|
| 国家基础信息 | 演示数据库 | 手动维护 |
| 景点信息 | SerpAPI (实时) + 演示数据库 | 混合 |
| 天气预报 | Open-Meteo API | 实时 |
| 汇率 | Exchange Rate API | 实时 |
| 签证信息 | 演示数据库 | 手动维护 |

## 5. 约束条件

### 5.1 技术约束

- 必须使用LangGraph构建多Agent系统
- 必须兼容原项目框架结构
- LLM调用使用SiliconFlow API
- 免费API优先

### 5.2 资源约束

- 开发时间：约2-3周
- 数据覆盖：优先支持5-10个热门国家
- API额度：使用免费额度

### 5.3 质量约束

- 代码必须有注释
- 遵循Python代码规范
- 必须可演示运行
